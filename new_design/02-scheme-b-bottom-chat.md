# 方案 B：底部控制台式 — Chat 作为悬浮控制台

## 1. 整体布局理念

**"Dashboard 是舞台，Chat 是导演用的对讲机，需要时抬头看，不需要时缩回底部。"**

这个方案保留 Dashboard 的完整全屏空间，将 Chat 压缩为底部一条悬浮输入条。用户的主要视觉焦点始终在 Dashboard 的内容上。点击输入条或发送消息后，Chat 历史以底部抽屉/右侧抽屉的形式展开。

这参考了 VS Code Copilot Chat、Discord 的输入体验、以及传统游戏开发引擎的底部控制台。

## 3. 区域详解

### 3.1 顶部全局栏
与方案 A 相同：返回按钮 + 项目名 + Build + Preview。

### 3.2 左侧：Chapter 时间线
与方案 A 相同：垂直手风琴树，Scene 带状态色标。

**额外优化**：在 Chapter 节点旁边增加快捷操作入口：
- Chapter 1 hover 时显示 `[▶]`（试玩本章）和 `[🤖]`（对 AI 说"重新生成本章"）
- 这样用户不展开底部 Chat 也能触发一些高频 AI 指令

### 3.3 中栏：主内容区
与方案 A 相同，标签页体系：
- 全局/Chapter 级别：`[Blueprint] [Story Map] [Audit Report]`
- Scene 级别：`[脚本] [资源] [审计]`

### 3.4 右栏：快捷操作面板
与方案 A 相同，但可以更宽一些（因为 Dashboard 占满了全宽）。

### 3.5 底部：悬浮输入条（核心创新点）

**这是整个方案的灵魂。**

一条约 56px 高的条，始终固定在屏幕底部：
- **左侧**：AI 头像图标（`🤖` 或机器人 icon），点击可展开 Chat 历史
- **中间**：单行输入框，placeholder 为"告诉 AI 下一步做什么..."
- **右侧**：发送按钮

**交互规则**：
1. 用户点击输入框或按下任意键 → 自动聚焦
2. 用户输入文字并回车/点击发送 → 根据内容决定展开形式：
   - 如果是简单指令（如"继续"）→ 可能只在底部条上方显示一个轻量回复气泡
   - 如果是复杂回复（如 BlueprintCard、AuditReportCard）→ 自动展开 Chat 抽屉
3. Chat 抽屉展开后，底部输入条**保持在抽屉底部**，而不是抽屉顶部（符合输入惯性）

## 4. 核心交互流程

### 流程 1：首次进入项目 → 查看并确认 Blueprint
1. 用户进入项目
2. Dashboard 显示 Blueprint
3. 底部输入条上方浮出 AI 提示气泡："已生成蓝图，请在 Dashboard 中查看"
4. 用户阅读 Blueprint 后，点击底部输入条，输入"确认，开始生成"
5. 发送后，底部 Chat 抽屉自动展开，显示 `BlueprintCard` 和确认反馈

### 流程 2：生成中 → 进度感知
1. 用户发送"开始生成第一章"
2. Chat 抽屉展开，显示 `ProgressCard`
3. 左侧 Chapter Timeline 同步更新状态
4. 用户可以关闭 Chat 抽屉，继续看 Dashboard，底部输入条始终可见
5. 当 Scene 生成完成或审计完成时，底部输入条上的 AI 头像可以有**角标通知**（红/黄/绿点），提示用户有重要消息

### 流程 3：审计失败 → 修复闭环
1. 审计完成，底部 AI 头像出现红色角标
2. 用户点击 AI 头像或输入条，Chat 抽屉展开
3. 显示 `AuditReportCard`（红色顶部）
4. 用户有两种选择：
   - 在 Chat 抽屉中点击"自动修复" → AI 修复
   - 点击"去 Dashboard 查看" → Dashboard 切到 Audit Report 标签
5. 修复完成后，AI 头像角标消失

### 流程 4：快速指令无需展开抽屉
1. 用户在底部输入条输入"重生成 Scene 1.2 的背景"
2. 发送后，AI 回复简短确认："已触发重生成"
3. 这个回复以**轻量 toast 气泡**的形式从底部输入条上方弹出，3 秒后消失
4. Chat 抽屉**不展开**，不打断用户在 Dashboard 的当前操作

## 5. Chat 抽屉的三种展开层级

| 层级 | 触发条件 | 高度/宽度 | 用途 |
|------|---------|----------|------|
| **L0: 仅输入条** | 默认状态 | 56px 高 | 快速输入简短指令 |
| **L1: 轻量回复** | AI 回复为简单文本或 Toast | 120-160px 高 | 显示简短反馈，不遮挡主内容 |
| **L2: 半高抽屉** | AI 回复包含 ProgressCard 或 AuditReportCard | 40-50% 屏幕高 | 显示结构化卡片，需要用户阅读 |
| **L3: 全高抽屉/右侧抽屉** | AI 回复包含 BlueprintCard 或复杂对话历史 | 右侧 420px 或底部 70% 高 | 深度审阅和决策 |

## 6. 优缺点

| 优点 | 缺点 |
|------|------|
| Dashboard 拥有最大可视空间，适合精细编辑脚本 | Chat 历史不常驻，需要时才能看到 |
| 底部输入条极其顺手，输入门槛最低 | 复杂的 Blueprint/Audit 卡片在底部抽屉中展示空间局促 |
| 不会挤压 Dashboard 的宽度，对笔记本友好 | 进度感知弱于方案 A，用户可能忘记看 Chat |
| 轻量回复不打扰，重内容才展开，节奏感好 | 需要依赖"AI 头像角标"来提醒用户查看重要消息 |

## 7. 设计 Agent 需重点绘制的页面

1. **默认收起状态**：底部只有输入条，Dashboard 展示 Blueprint
2. **轻量回复状态**：底部输入条上方浮出 AI toast（如"已触发重生成"）
3. **半高抽屉状态**：底部升起抽屉，内有 `ProgressCard` 和 `AuditReportCard`
4. **右侧抽屉变体**：点击 AI 头像后右侧 Chat Drawer 展开
5. **带角标通知的输入条**：AI 头像有红色/黄色通知点
6. **Chapter Timeline hover 状态**：显示快捷操作 `[▶]` 和 `[🤖]`

## 8. 与方案 A 的核心差异

| | 方案 A（左右分屏） | 方案 B（底部控制台） |
|---|---|---|
| Chat 可见性 | 始终可见（右侧 35%） | 默认收起，需要时展开 |
| Dashboard 空间 | 被压缩 | 全屏 |
| 输入门槛 | 低（始终有输入框） | 极低（底部悬浮条） |
| 适合场景 | 高频对话、重度 AI 协作 | 高频编辑、间歇性对话 |
| 进度感知 | 极强（左右联动同时可见） | 中等（依赖状态更新+通知角标） |
