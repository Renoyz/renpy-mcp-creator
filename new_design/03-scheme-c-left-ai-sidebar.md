# 方案 C：左侧 AI 边栏式 — Chat 作为创意助理侧边栏

## 1. 整体布局理念

**"AI 是坐在你左手边的创意助理，随时待命；你正对着主工作台（Dashboard）创作，需要时转头看一眼。"**

这个方案将 ChatPanel 放在**左侧**，Dashboard 主工作区放在**右侧**。这与方案 A 的左右关系恰好相反，也与方案 B 的底部悬浮完全不同。

其心智模型参考了：
- Notion AI（左侧/顶部触发，但内容在右侧主区域）
- Replit Ghostwriter（侧边 AI 助手）
- 传统设计软件的"左侧面板"（如 Figma 的左侧面板是图层和组件，右侧是属性）

在这里，**左侧 = AI 对话 + Chapter 时间线**，**右侧 = 内容编辑主舞台**。

## 3. 区域详解

### 3.1 左侧边栏：上下两区（核心创新点）

左侧不是单一功能，而是**上下分割为两个面板**：

#### 上区：ChatPanel（占左侧高度的 60-70%）
- 顶部：项目名称 + 设置按钮
- 中部：消息流（与方案 A/B 相同，支持 BlueprintCard、ProgressCard、AuditReportCard 等）
- 底部：输入框 + 发送按钮

#### 下区：Chapter Timeline（占左侧高度的 30-40%）
- 可折叠/可调整高度（拖拽分割线）
- 显示项目的 Chapter/Scene 手风琴树
- Scene 带状态色标

**为什么把 Chapter Timeline 和 Chat 放在同一侧？**
- 因为两者都是"导航/指挥"性质的功能，不是"编辑"性质的功能。
- 用户在左侧"发指令+看进度"，在右侧"审阅内容+做修改"，动作分工非常清晰。
- 右侧主舞台完全不被任何导航元素打扰，适合沉浸式阅读和编辑。

### 3.2 右侧：Dashboard 主舞台（全宽、无内部分栏）

右侧不采用三栏，而是**单一大舞台**，通过标签页切换不同视图：

**全局/项目级别视图**：
- **Blueprint 标签**（默认）：展示完整项目蓝图
- **Story Map 标签**：React Flow 故事图，全宽展示
- **Audit Report 标签**：审计报告全页展示

**Scene 级别视图**：
- **脚本标签**：Monaco 编辑器，占满右侧主舞台
- **资源标签**：该 Scene 的所有资源大图展示
- **审计标签**：该 Scene 的审计详情

**快捷操作的处理**：
- 不像方案 A 那样单独设右栏，而是将快捷操作**嵌入在内容区内部**：
  - Blueprint 页面右上角有 `[修改 Blueprint]`、`[开始生成]` 按钮
  - 脚本编辑器右上角有 `[试玩本章]`、`[重生成背景]` 按钮
  - 资源页面每个图片卡片上有 `[重生成]` 按钮

### 3.3 顶部全局栏
- **左侧**：返回项目列表按钮 ←
- **中间**：当前项目名称
- **右侧**：`[Build]` + `[Preview]`

## 4. 核心交互流程

### 流程 1：首次进入项目 → 确认 Blueprint
1. 用户进入项目
2. 右侧主舞台显示 Blueprint 标签（全宽大图）
3. 左侧 ChatPanel 显示 `BlueprintCard`："请确认项目蓝图"
4. 用户在右侧阅读完整的 Blueprint（角色、章节、基调）
5. 用户在左侧 ChatPanel 点击"确认并生成"
6. Orchestrator 开始生成 Chapter 1

### 流程 2：生成中的进度感知
1. 用户在左侧 Chat 发送"开始生成第一章"
2. 左侧 ChatPanel 显示 `ProgressCard`
3. 左侧下部的 Chapter Timeline 同步更新 Scene 状态
4. 右侧主舞台**保持当前视图不变**（比如用户正在看 Blueprint，可以一边看大纲一边等生成）
5. 当某个 Scene 生成完成后，如果用户当前在 Blueprint 标签，该 Scene 卡片状态自动变绿

### 流程 3：选中 Scene 进行编辑
1. 用户在左侧下部的 Chapter Timeline 中点击"Scene 1.2"
2. 右侧主舞台切换到"脚本"标签，加载 scene_1_2.rpy
3. Monaco 编辑器占满右侧，给用户最大的编辑空间
4. 编辑器工具栏上显示该 Scene 的快捷操作：`[试玩]`、`[重生成背景]`、`[重生成角色]`

### 流程 4：审计失败后的修复
1. 审计完成，左侧 ChatPanel 显示 `AuditReportCard`（红色）
2. 用户点击"查看详情"
3. 右侧主舞台切换到"Audit Report"标签
4. 报告全页展示，issue 列表在左，详情/建议在右（或上下布局）
5. 用户点击某条 issue 的"打开脚本"
6. 右侧主舞台切到"脚本"标签，自动定位到对应行
7. 用户修改后保存，回到左侧 Chat 发送"重新审计"

### 流程 5：从 Story Map 跳转编辑
1. 用户在右侧主舞台切换到"Story Map"标签
2. 全宽 React Flow 图展示整个项目结构
3. 用户点击 Chapter 1 的某个节点
4. 左侧下部的 Chapter Timeline 自动展开并高亮对应 Scene
5. 右侧主舞台自动切换为该 Scene 的"脚本"标签

## 5. 左侧边栏的折叠机制

为了兼容不同屏幕尺寸和用户使用习惯，左侧边栏支持两种模式：

### 模式 1：展开模式（默认，宽屏）
- 左侧占屏幕宽度 25-30%
- Chat 和 Chapter Timeline 同时可见

### 模式 2：收缩模式（窄屏或用户手动收起）
- 左侧收缩为一条 48px 宽的窄边栏
- 显示两个图标：`💬`（展开 Chat）和 `📑`（展开 Chapter Timeline）
- 点击图标后，对应面板以**悬浮抽屉**形式从左侧滑出
- 这类似于 Slack 的左侧面板或 Figma 的左侧面板收起态

```
收缩模式：
┌──┬────────────────────────────────────────────────────────────────┐
│💬│ ← 返回    campus_romance                            [Build][Play]│
│📑├────────────────────────────────────────────────────────────────┤
│  │                                                                │
│  │              Dashboard 主舞台（占满剩余空间）                    │
│  │                                                                │
│  │                                                                │
│  │                                                                │
└──┴────────────────────────────────────────────────────────────────┘
```

## 6. 优缺点

| 优点 | 缺点 |
|------|------|
| 右侧主舞台空间最大，适合沉浸式编辑和全宽 Story Map | Chat 不占据右侧黄金位置，可能降低对话感知 |
| 左侧面板同时承载 Chat 和 Timeline，信息密度高 | 左侧上下分割可能让小屏幕用户感到拥挤 |
| 创作动作（左侧）和编辑动作（右侧）物理分离，不易混淆 | 如果用户长时间不看左侧，可能错过 Chat 消息 |
| 主内容区无内部分栏，视觉简洁，适合中文排版 | Audit Report 的快捷操作需要嵌入内容区，不如右栏直观 |
| 对右撇子友好：主要编辑区域在右侧，鼠标移动路径自然 | 需要较强的"左发指令、右做编辑"的用户教育 |

## 7. 与方案 A、B 的核心差异

| 维度 | 方案 A（Chat 在右） | 方案 B（Chat 在底） | 方案 C（Chat 在左） |
|------|-------------------|-------------------|-------------------|
| Chat 位置 | 右侧固定分屏 | 底部悬浮/抽屉 | 左侧固定边栏 |
| Dashboard 空间 | 被压缩至 65% | 全屏 | 被压缩至 70-75% |
| 进度导航位置 | 左栏 | 左栏 | 左下区（与 Chat 同侧） |
| 主编辑区位置 | 中栏（被三栏分割） | 全屏主内容区 | 右栏（单一大舞台） |
| 最适合的用户 | 重度 AI 对话用户 | 间歇对话+重度编辑用户 | 需要大编辑空间+结构化导航的用户 |
| 参考产品 | ChatGPT Canvas, Cursor | VS Code Copilot, Discord | Notion, Replit, Figma |

## 8. 设计 Agent 需重点绘制的页面

1. **默认展开状态**：左侧上下分栏（Chat 上 + Timeline 下），右侧全宽 Blueprint
2. **编辑脚本状态**：左侧 Timeline 高亮 Scene 1.2，右侧全宽 Monaco 编辑器
3. **审计报告状态**：左侧 Chat 显示红色 `AuditReportCard`，右侧全宽 Audit Report 页面
4. **Story Map 状态**：右侧全宽 React Flow 图，左侧 Timeline 同步高亮
5. **生成中状态**：左侧 Chat 的 `ProgressCard` + Timeline 的 `🔄 generating` 动画
6. **收缩模式**：左侧变为 48px 图标边栏，主舞台占满

## 9. 特殊变体：混合模式（推荐作为默认）

方案 C 还有一个灵活的变体：
- **默认展开 Chapter Timeline**：左侧下部始终显示 Timeline
- **Chat 以"对话气泡堆叠"形式存在于 Timeline 上方**：不是完整的消息流，而是只显示最近 2-3 条关键卡片
- 用户需要查看完整对话历史时，点击"展开完整对话"，Chat 才占满左侧上部

```
混合模式默认态：

│  🤖 AI 协作         │ ← 返回    campus_romance                [Build][Play]│
│                    ├─────────────────────────────────────────────────────┤
│  ┌─────────────┐   │                                                     │
│  │ Blueprint   │   │              Dashboard 主舞台                        │
│  │ Card（最近） │   │                                                     │
│  └─────────────┘   │                                                     │
│  [输入消息...]      │                                                     │
├────────────────────┤                                                     │
│  Chapter Timeline  │                                                     │
│  ▼ Ch 1            │                                                     │
│   🟢 1.1           │                                                     │
│   🟢 1.2           │                                                     │
│   🔴 1.3           │                                                     │
│  ▶ Ch 2            │                                                     │
└────────────────────┴─────────────────────────────────────────────────────┘
```

这个变体的好处是：左侧始终给 Timeline 留出足够空间，Chat 只显示最关键的信息，避免上下拥挤。
