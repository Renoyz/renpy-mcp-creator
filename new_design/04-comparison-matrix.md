# 方案对比矩阵 — 供设计 Agent 快速决策

## 1. 三种方案鸟瞰

| 方案 | 命名 | Chat 位置 | Dashboard 空间 | 进度导航位置 | 核心心智模型 |
|------|------|-----------|---------------|-------------|-------------|
| **A** | 左右分屏式 | 右侧固定（35%） | 左侧 65% | 左栏 | Chat 和 Dashboard 是双主角，并列协作 |
| **B** | 底部控制台式 | 底部悬浮（可展开抽屉） | 全屏 | 左栏 | Dashboard 是舞台，Chat 是导演对讲机 |
| **C** | 左侧 AI 边栏式 | 左侧固定（25-30%） | 右侧 70-75% | 左下区 | 左侧是 AI 助理，右侧是主工作台 |

## 2. 关键使用场景下的表现

### 场景 1：高频对话 + 轻度审阅（如确认 Blueprint、下达生成指令）
| 方案 | 评分 | 原因 |
|------|------|------|
| A | ⭐⭐⭐⭐⭐ | Chat 始终可见，无需任何点击即可对话 |
| B | ⭐⭐⭐⭐☆ | 底部输入条始终可见，但复杂卡片需要展开抽屉 |
| C | ⭐⭐⭐⭐☆ | Chat 固定可见，但位于左侧边缘，注意力稍弱 |

### 场景 2：沉浸式脚本编辑（如修改台词、调整缩进）
| 方案 | 评分 | 原因 |
|------|------|------|
| A | ⭐⭐⭐☆☆ | Dashboard 被压缩，编辑器宽度受限 |
| B | ⭐⭐⭐⭐⭐ | Dashboard 全屏，编辑器空间最大 |
| C | ⭐⭐⭐⭐☆ | 主舞台占 70-75%，空间充足 |

### 场景 3：进度感知（生成中频繁看"到哪了"）
| 方案 | 评分 | 原因 |
|------|------|------|
| A | ⭐⭐⭐⭐⭐ | 左右联动同时可见，感知最强 |
| B | ⭐⭐⭐☆☆ | 依赖 Timeline 动画+底部角标，需要主动关注 |
| C | ⭐⭐⭐⭐☆ | Timeline 在左下固定可见，但离主编辑区较远 |

### 场景 4：审计报告审阅（issue 多、涉及多文件）
| 方案 | 评分 | 原因 |
|------|------|------|
| A | ⭐⭐⭐⭐⭐ | Audit Report 可作为 Dashboard 主标签全宽展示，同时 Chat 保留上下文 |
| B | ⭐⭐⭐☆☆ | Audit Report 全宽展示好，但 Chat 抽屉可能遮挡 |
| C | ⭐⭐⭐⭐☆ | 主舞台全宽展示 Audit Report，但左侧 Chat 和 Timeline 分担注意力 |

### 场景 5：小屏幕笔记本（13-14 寸）
| 方案 | 评分 | 原因 |
|------|------|------|
| A | ⭐⭐☆☆☆ | 35% 给 Chat 后 Dashboard 过窄，体验局促 |
| B | ⭐⭐⭐⭐⭐ | 全屏 Dashboard + 可收起 Chat，最省空间 |
| C | ⭐⭐⭐⭐☆ | 左侧 25% 相对克制，主舞台仍有足够空间 |

## 3. 设计 Agent 的选择建议

### 如果只能选一个主推方案：
- **桌面端主力用户多、屏幕尺寸分布均匀** → **推荐方案 A**
- **用户以笔记本为主、编辑时间远多于对话时间** → **推荐方案 B**
- **追求简洁大气、强调内容舞台感** → **推荐方案 C**

### 更现实的工程选择：
**以方案 A 为桌面端默认，以方案 B 为窄屏/移动端降级。**

原因：
- 方案 A 最符合"AI 生成工具"的产品定位，Chat 的核心地位得到了视觉上的尊重。
- 方案 B 的底部控制台在宽度不足时自然替代方案 A，无需重新设计交互逻辑。
- 方案 C 虽然优雅，但"左指令右编辑"的心智模型需要较强的用户教育成本。

## 4. 通用组件清单（三方案共享）

无论选择哪个方案，以下组件都需要设计：

### 4.1 Chat 消息卡片
- `BlueprintCard`：角色列表 + Chapter 网格 + 确认/修改按钮
- `ProgressCard`：进度条 + 当前步骤描述 + 预计时间
- `AuditReportCard`：状态色块 + issue 折叠列表 + 操作按钮
- `ResourceCandidateCard`：2 列图片网格 + 单张确认/重生成按钮
- `TextMessageCard`：普通文本气泡

### 4.2 Dashboard 页面
- `BlueprintOverview`：Cast 角色卡 + Scenes 网格 + 叙事基调
- `SceneDetail`：脚本编辑器（Monaco）+ 资源画廊 + 审计记录
- `StoryMapGraph`：React Flow 只读故事图
- `AuditReportPage`：五维 issue 分组展示 + 跳转修复

### 4.3 导航/状态组件
- `ChapterTimeline`：垂直手风琴树 + Scene 状态色标
- `GlobalHeader`：返回按钮 + 项目名 + Build + Preview
- `SceneStatusBadge`：`pending`/`generating`/`confirmed`/`audit_fail` 的标识

## 5. 需要避免的设计陷阱

| 陷阱 | 说明 | 规避方式 |
|------|------|---------|
| **把 Chat 当成功能按钮** | 顶部栏放一个"AI 助手"按钮，Chat 作为抽屉 | Chat 必须有与其核心地位匹配的位置（常驻侧边或底部） |
| **Blueprint 没有独立页面** | Blueprint 只出现在 Chat 消息中，Dashboard 没有入口 | Dashboard 默认视图或独立标签必须能展示完整 Blueprint |
| **审计结果埋没在消息流** | AuditReport 只以文本形式出现在 Chat 中 | 必须有独立的 Audit Report 页面/标签 |
| **iframe 割裂体验** | Story Map / 编辑器 / 资源库各自是 iframe | 全部改为内联 React 组件，支持联动跳转 |
| **状态色标含义混乱** | 不同页面的 Scene 状态用不同颜色体系 | 建立唯一的 `SceneStatus` 设计系统 |

## 6. 交付建议优先级

**Tier 1（必须设计）**：
1. 项目驾驶舱默认态（含 Chapter Timeline + Blueprint 视图）
2. Chat 的 `BlueprintCard` 和 `AuditReportCard`
3. Scene 编辑态（脚本编辑器 + 资源标签）
4. Audit Report 全页视图

**Tier 2（强烈建议）**：
5. Story Map 全页视图
6. 生成中状态（ProgressCard + Timeline 动画）
7. 审计失败修复流程

**Tier 3（可选）**：
8. 移动端/窄屏变体
9. 深色/浅色主题对比
10. 空状态/加载状态
