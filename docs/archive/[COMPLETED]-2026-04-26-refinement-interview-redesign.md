# 需求澄清访谈改进方案

## 当前问题

当前"访谈"并非 LLM 驱动的自适应对话，而是硬编码的 2 轮固定提问：

```
用户触发 → 第1轮固定问题 → 第2轮固定问题 → LLM一次性生成蓝图 → 结束
```

| 问题 | 详情 |
|------|------|
| 无自适应追问 | 用户说"悬疑"，系统不问"本格？社会派？惊悚？" |
| 被动等答案，不给选项 | 用户不知道怎么写视觉风格，系统只会问"你想要什么风格？"——用户卡住 |
| 无逐槽填充 | 9 个 slot 靠 LLM 从对话后期"扒"出来，不是一问一答填满的 |
| 遗漏关键维度 | 视觉风格、音频、分支结构、结局数、平台限制、内容分级、参考作品——从未被问及 |
| 无迭代机制 | 用户拒绝蓝图 = 全部重置，不能"保留角色但换设定" |
| 章节大纲是字符串拼接 | `chapter_goal = f"Advance {chapter.name} through {first_scene}"` —— 不是设计 |

---

## 核心设计理念：从"提问者"变为"创意搭档"

当前模式：**被动提问 → 等待用户给出确定答案 → 填充槽位**

```
系统: "你想要什么视觉风格？"
用户: "呃……我不太确定"          ← 卡住了
系统: "请回答，这很重要"          ← 死循环
```

改进模式：**主动提供备选方案 → 用户选择/混合 → 细化 → 确认**

```
系统: "基于你的明朝悬疑题材，这里有三条风格路线：
       A. 水墨暗调 — 大面积留白，墨色晕染，适合营造迷雾感
       B. 工笔重彩 — 细节精密，色彩浓郁，适合表现服饰器物之美
       C. 浮世绘风 — 平面装饰感，大胆构图，适合动作场景的张力
      你觉得哪个方向更接近你要的感觉？也可以混搭。"

用户: "A 和 B 的结合吧，水墨的暗调但保留工笔的细节"  ← 从选项中获得了方向
```

### 三条行为准则

**准则 1：先提案，再收束**

对任何空槽位，不要直接问"你想要什么"。先基于已有信息（题材、世界观、已确定的角色）生成 2-4 个有区分度的备选方案，让用户在选项之间做出判断，而不是凭空想象。

**准则 2：允许"我不知道"，主动补位**

用户说"我不确定"、"没想好"是合理状态。此时主动基于同题材成熟作品给出推荐方案，把决策从"开放式创作"降维成"选择题"。

**准则 3：交叉一致性检查**

每确定一个槽位后，检查是否与其他已确定槽位矛盾。例如用户选了"水墨暗调"但之前说"色彩鲜艳的童话世界"——主动指出矛盾，给出两种化解路径让用户选择。

用户说"我想做一个明朝背景的悬疑冒险，锦衣卫调查海外岛屿的诡异事件"。

系统不应直接问"还有什么要补充的吗"，而应给出：

```

│ 核心冲突：朝廷内部的权力斗争通过岛屿秘密浮出水面        │

┌─ B. 克苏鲁奇幻 ──────────────────────────────────┐
│ 世界观：郑和带回的不仅有宝物，还有不该被唤醒的存在       │
│ 核心冲突：人类文明 vs 不可名状的远古力量              │
│ 基调：理智逐渐剥落，从理性探索坠入疯狂                 │
│ 参考：《诡秘之主》×《克苏鲁的呼唤》                   │
```

## 实现方案

### 架构变更

将当前 `BlueprintOrchestrator` 的硬编码 turn-by-turn 逻辑替换为 **LLM 驱动的创意搭档访谈循环**。

**当前架构：**
```
handle_user_message()
  ├─ turn_count == 0 → 返回硬编码问题A
  ├─ turn_count == 1 → 返回硬编码问题B
  └─ turn_count >= 2 → _generate_draft_via_llm() 一次性生成
```

**改进后架构：**
```
handle_user_message()
  └─ _conduct_interview_round()
       ├─ 1. 组装上下文: 已填槽位 + 空缺槽位 + 对话历史
       ├─ 2. LLM 创意搭档调用: 系统提示 + 槽位状态 → LLM 返回响应
       │    响应类型: 备选方案提案 | 追问澄清 | 一致性确认 | 阶段小结
       ├─ 3. 解析响应: 提取 <OPTIONS> 提案块 + <META> 槽位更新
       ├─ 4. 追踪提案历史: 记录"给了什么选项、用户选了什么"
       ├─ 5. 判断: 必填槽位全部填满? → 最终确认 → 生成蓝图 : 继续
       └─ 6. 返回 LLM 的选项/问题给用户
```

### 响应格式设计

LLM 的每条响应可以包含多种功能块，后端解析但不展示标签：

```
<PHASE>phase_2_tone_style</PHASE>

<OPTIONS id="visual_style_opts">
## 视觉风格方案

基于你的明朝悬疑题材，给出三条路线：
┌─ A. 水墨暗调 — ...
└─ B. 工笔重彩 — ...
└─ C. 浮世绘风 — ...
</OPTIONS>

<QUESTION>
你倾向哪个方向？还是想混搭？
</QUESTION>

<META>
{
  "phase": "tone_style",
  "proposals": [
    {"id": "visual_style_opts", "for_slot": "visual_style",
     "options": ["A. 水墨暗调", "B. 工笔重彩", "C. 浮世绘风"]}
  ],
  "slot_updates": {}
}
</META>
```

用户选择后：
```
用户: "A 和 B 结合，水墨的暗调但保留工笔的细节"

→ 后端解析到这是对 visual_style_opts 的回应
→ 更新 slot visual_style = "水墨暗调与工笔重彩结合：...”
→ 标记提案 visual_style_opts 为 selected
```

### 提案追踪

后端维护一个 `proposal_history`，记录 LLM 每次给了什么选项、用户选了哪个：

```python
proposal_history = [
    {
        "proposal_id": "visual_style_opts",
        "for_slot": "visual_style",
        "options": ["A. 水墨暗调", "B. 工笔重彩", "C. 浮世绘风"],
        "user_choice": "A+B 混合",
        "resolved_slot_value": "水墨暗调与工笔重彩结合：..."
    }
]
```

这个数据用于：
1. 如果用户后来想改，可以回到这个决策点
2. 最终生成时标注哪些是"用户指定"，哪些是"系统推荐未被修改"
3. 帮助一致性检查：后续选择是否与此选择矛盾

### 访谈系统提示词设计

```
你是视觉小说项目的创意搭档（creative partner），不是调查问卷填写员。

## 你的核心职责
帮助作者把模糊的想法变成具体的、可执行的游戏企划。
你不是在"收集信息"——你是在"与作者一起创造"。

## 你的行为准则

### 准则 1: 主动提案
对任何空缺槽位，不要直接问"你想要什么"。而是：
- 基于已确定的内容（题材、世界观、角色）生成 2-4 个有区分度的备选方案
- 每个方案要有独特的定位和参考作品
- 方案之间要有足够的差异让用户能做出有意义的判断
- 用 <OPTIONS> 标签包裹提案内容

### 准则 2: 降维选择
如果作者说"不确定""没想好""随便"：
- 主动给出基于该题材最成功的 2-3 种常规做法
- 标注你的推荐（标注理由）
- 把创作决策从"开放式自由创作"降维成"选择题"

### 准则 3: 交叉检查
每次更新槽位后，检查与其他已填槽位的逻辑一致性。
如果发现矛盾，主动指出并给出两种化解路径。

### 准则 4: 节奏控制
- 每次只提 1 个话题，给 2-4 个选项
- 不要在一条消息里同时讨论视觉风格和角色关系
- 用户做出选择后，先确认理解正确，再进入下一个话题

### 准则 5: 溯源可溯
- 每个槽位的值标注来源: "user_specified" 还是 "system_recommended"
- 如果用户直接给出了明确答案，不要强行再提案
- 如果用户接受了系统推荐的选项未被修改，标记为 system_recommended

## 需要填充的槽位
系统会在每条消息中附带当前槽位状态。

## 完成标准
当所有必填槽位都已填充且无矛盾时，输出汇总确认（标注每个项的来源），
用户确认后回复 <CONCLUSION>。

## 你绝对不能
- 在作者没参与的情况下替作者做决定
- 只给一个选项（那不是选择）
- 跳过必填槽位
- 在首次提案时就假设角色数量和章节数量——先让用户确认
```

每个槽位不仅记录值，还记录来源：

```
Current Slots:
  ✅ core_premise: "明初郑和时代，锦衣卫调查..." (user_specified)
  ✅ audience_genre: "架空历史/悬疑冒险" (user_specified)

  vs_001: "A.水墨/B.工笔/C.浮世绘" → pending
  char_001: "A.忠犬/B.倦怠/C.暗面" → selected B

Next priority: resolve visual_style (proposal pending), then tone_themes
```

### 访谈终止条件

1. **所有必填槽位已填满且用户确认** → LLM 展示汇总 → 用户确认 → `<CONCLUSION>`
2. **用户主动结束** → "可以了，直接生成" → 用现有数据生成
3. **轮次上限** → 最多 25 轮（比纯被动模式多 5 轮，因为提案→选择一轮变两轮），达到上限时强制汇总生成
4. **用户要求修改已填槽位** → 回到对应 Phase，只为该槽位重新提案，不影响其他

### 向后兼容

- `_START_TRIGGERS` 和 `_INTAKE_START_TRIGGERS` 保持不变
- `RefinementIntake` 数据结构不变（新增 `source` 字段可选）
- 后续的 brief 确认 UI 流程不变
- `_generate_draft_via_llm()` 在访谈结束后做最终整合

### 回退方案

如果 LLM 访谈响应解析失败或提案循环卡死（同一槽位重复提案 >3 次），回退到硬编码 2 轮访谈。保留旧代码路径。

## 验证方式

1. 用 `start_refinement_intake` 触发新访谈
2. 给模糊初始想法"想做悬疑"，验证 LLM 给出 3 条世界观路线，每条有区分度
3. 说"不确定"，验证 LLM 标出推荐选项并给出理由
4. 选择混合方案"A 的基调 B 的敌人"，验证 LLM 正确合并
5. 中途说"收束到选项 A"，验证该槽位标记为 user_specified
6. 全部填满后，验证汇总中每个项标注了来源
7. 运行现有测试确保不回归
